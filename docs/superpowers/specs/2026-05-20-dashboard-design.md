# 数据看板系统 (Dashboard) 设计文档

**日期**: 2026-05-20
**状态**: 已确认，待实现

---

## 1. 概述

为 price-monitor 系统新增一个综合数据看板页面 (`/dashboard`)，将商品监控、职位监控和系统运营数据以可视化的方式集中呈现。看板支持实时数据更新，根据用户角色动态展示不同内容，并适配桌面端和移动端。

---

## 2. 设计决策

| 维度 | 决策 |
|---|---|
| 目标用户 | 混合模式——普通用户看个人指标，管理员额外看系统运营指标 |
| 页面定位 | 新建独立 `/dashboard` 路由，从导航栏进入 |
| 刷新策略 | SSE 实时推送 KPI 变化，趋势图按需 HTTP 查询 |
| 核心 KPI | 全量覆盖：商品监控 + 职位监控 + 系统运营 |
| 图表库 | ECharts |
| 时间范围 | 可切换：7天 / 30天 / 90天 |
| 布局结构 | 分栏——左侧/上半个人指标区，右侧/下半管理员系统运营区 |
| 管理员区块 | 普通用户完全不可见 |
| 移动端 | 必须完全适配，图表自动调整为单列 |
| 数据架构 | 混合方案——Redis 缓存 KPI（每分钟刷新），趋势图走 PostgreSQL 聚合查询 |
| KPI 实时性 | 包含实时指标（如今日爬取次数、今日降价数） |
| 个人区组织 | 商品区块在上，职位区块在下，上下排列 |

## 3. 整体架构

```
┌─────────────┐     SSE      ┌─────────────────┐
│  Dashboard  │◄────────────►│  FastAPI SSE    │
│    Page     │              │  /api/dashboard/│
│  (React)    │              │    events       │
└──────┬──────┘              └────────┬────────┘
       │                              │
       │  HTTP GET /api/dashboard/    │ 每分钟比较
       │  trends?type=x&days=N        │ KPI 变化
       │                              │
       ▼                              ▼
┌──────────────────────────────────────────┐
│              FastAPI Backend              │
│  ┌─────────────┐    ┌─────────────────┐  │
│  │ KPI Service │───►│  Redis Cache    │  │
│  │ (预聚合)    │    │ dashboard:kpi:* │  │
│  └─────────────┘    └─────────────────┘  │
│  ┌─────────────┐    ┌─────────────────┐  │
│  │ Trend       │───►│  PostgreSQL     │  │
│  │ Service     │    │  (按需聚合)     │  │
│  │ (按需查询)  │    └─────────────────┘  │
│  └─────────────┘                          │
└──────────────────────────────────────────┘
```

### 3.1 数据流说明

1. **KPI 实时更新流**：前端建立 SSE 连接 → 后端每分钟从 Redis 读取 KPI 数据 → 与上次推送值比较 → 有变化时推送给前端 → 前端更新 KPI 卡片
2. **趋势图数据流**：用户切换时间范围或进入页面 → 前端发送 HTTP GET 请求 → 后端执行 PostgreSQL 聚合查询 → 返回趋势数据 → 前端渲染 ECharts 图表
3. **后端聚合任务**：每分钟定时任务执行 `refresh_dashboard_stats()` → 查询 PostgreSQL 聚合数据 → 写入 Redis

## 5. 区块详细设计

### 5.1 KPI 卡片行（个人区）

| 卡片 | 指标 | 实时性 | 数据来源 |
|---|---|---|---|
| 监控商品数 | `COUNT(products)` | 静态 | products 表 |
| 今日降价数 | 今日价格低于上次记录的商品数 | 实时 | price_history 表 |
| 新职位数 | 今日首次出现的职位数 | 实时 | jobs 表 |
| 匹配分析数 | 今日完成的匹配分析数 | 实时 | job_match 表 |
| 今日爬取次数 | 今日所有爬取操作总数 | 实时 | crawl_logs + job_crawl_logs |

**交互**：
- 数字变化时带有动画效果（数字滚动）
- 卡片可点击跳转到对应详情页（如"今日降价数"跳转到商品列表）

### 5.2 商品监控区块

**各平台商品分布（饼图）**
- 维度：平台（淘宝 / 京东 / 亚马逊）
- 数据：`COUNT(products) GROUP BY platform`

**价格变动趋势（折线图）**
- X轴：日期（按天聚合）
- Y轴：平均价格 / 价格变动次数
- 维度：可切换显示"平均价格趋势"或"降价次数趋势"
- 数据：按 `scraped_at` 聚合 `price_history`

### 5.3 职位监控区块

**各平台职位分布（饼图）**
- 维度：平台（boss / 51job / 猎聘）
- 数据：`COUNT(jobs) GROUP BY platform`

**新增职位趋势（折线图）**
- X轴：日期（按天聚合）
- Y轴：每日新增职位数
- 数据：按 `first_seen_at` 聚合 `jobs`

**薪资区间分布（柱状图）**
- X轴：薪资区间（如 0-5k, 5-10k, 10-15k, 15-20k, 20k+）
- Y轴：职位数量
- 数据：按 `salary_min` 分组统计 `jobs`

### 5.4 系统运营区（仅 admin）

**系统 KPI 卡片**
- 总用户数：`COUNT(users)`
- 总爬取次数：今日所有爬取操作总数
- 系统成功率：`success / total`
- 活跃告警数：`COUNT(alerts) WHERE resolved = false`
- 磁盘使用率：`psutil.disk_usage('/').percent`
- 内存使用率：`psutil.virtual_memory().percent`

**系统健康度趋势（折线图）**
- X轴：时间（按小时聚合）
- Y轴：成功率百分比
- 多条线：整体成功率、商品爬取成功率、职位爬取成功率

**平台成功率对比（柱状图）**
- X轴：平台（淘宝 / 京东 / 亚马逊 / boss / 51job / 猎聘）
- Y轴：成功率百分比
- 数据：按 platform 聚合 crawl_logs / job_crawl_logs

**最近告警列表（表格）**
- 列：时间、类型、内容、状态
- 数据：最近的告警记录，最多显示 10 条

## 6. API 设计

### 6.1 SSE 端点

```
GET /api/dashboard/events
Content-Type: text/event-stream

Response (SSE):
event: kpi_update
data: {
  "total_products": 42,
  "price_drops_today": 3,
  "new_jobs_today": 15,
  "match_count": 8,
  "crawl_count_today": 120
}

event: system_kpi

  "total_users": 10,
  "total_crawls": 5000,
  "success_rate": 0.95,
  "active_alerts": 2,
  "disk_usage": 0.72,
  "memory_usage": 0.45

```

```
GET /api/dashboard/trends?type={type}&days={days}

  type: "price" | "jobs" | "salary" | "system_health" | "platform_success"

      "data": [100.5, 99.8, ...]
    }
  ]
}
```

## 7. 后端实现

### 7.1 Redis 缓存结构

```
dashboard:kpi:{user_id}  (Hash)
  - total_products: int
  - price_drops_today: int
  - new_jobs_today: int
  - match_count: int
  - crawl_count_today: int
  - last_updated: timestamp

dashboard:kpi:system  (Hash)
  - total_users: int
  - total_crawls: int
  - success_rate: float
  - active_alerts: int
  - disk_usage: float
  - memory_usage: float
  - last_updated: timestamp
```

### 7.2 定时聚合任务

每分钟执行 `refresh_dashboard_stats()`：

1. 遍历所有活跃用户，计算个人 KPI 写入 Redis
2. 计算系统级 KPI 写入 Redis
3. 记录聚合耗时日志

### 7.3 SSE 推送逻辑

1. 客户端连接 `/api/dashboard/events`
2. 立即推送当前全量 KPI
3. 启动每分钟定时器：
   - 读取 Redis 中的 KPI 数据
   - 与上次推送值比较
   - 有变化时推送增量更新
4. 连接断开时清理定时器

## 8. 错误处理

| 场景 | 处理策略 |
|---|---|
| Redis 不可用 | 降级为直接查询 PostgreSQL（慢但可用），记录降级事件 |
| SSE 连接断开 | 前端自动重连（指数退避），重连期间显示"连接中"状态 |
| 聚合查询超时 | 图表区域显示"数据加载超时"，提供"重试"按钮 |
| 权限越界 | 普通用户访问管理员 API 返回 403，前端不渲染对应区块 |
| 系统资源监控失败 | 显示"N/A"，不影响其他指标展示 |

## 9. 前端实现要点

### 9.1 技术栈
- React + TypeScript + ECharts (via `echarts-for-react`)
- Ant Design 组件（Card、Tag、Table 等）
- 响应式布局：桌面端分栏，移动端单列

### 9.2 SSE 连接管理
- 使用 `EventSource` API
- 页面挂载时建立连接，卸载时关闭
- 自动重连机制：1s → 2s → 4s → 8s → 最大 30s

### 9.3 角色条件渲染
- 通过 `useAuth()` 获取当前用户角色
- `role === 'admin'` 时渲染系统运营区
- API 层面也做权限校验（双重保险）

### 9.4 响应式断点
- 桌面端：`>= 1024px`，分栏布局
- 平板端：`768px - 1023px`，单列但图表较大
- 移动端：`< 768px`，单列紧凑布局
